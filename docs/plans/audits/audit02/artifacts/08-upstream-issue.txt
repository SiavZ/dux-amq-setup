Title:  Native HMAC envelope for inter-agent auth (proposed protocol)
Labels: enhancement, security, wake/inject
------------------------------------------------------------------------

Summary
-------
AMQ has no built-in authentication between peers on the file-based
queue. A pane that can write to `/data/state/amq/<victim>/inbox/` can
spoof `--me <other>` and inject arbitrary text into the victim agent's
TTY through `amq wake --inject-mode raw`. We need a way for receivers
to verify the body originated from a trusted sender before injection.

The dux-amq overlay implements a wrapper-side mitigation in audit02
phase 08 (`amq-send-signed`, `amq-receive-verify`, scripts/) that wraps
each body in a `DUX1\t<from>\t<to>\t<ts>\t<nonce>\t<mac>\t<body>`
envelope signed with HMAC-SHA256 over `from|to|ts|nonce|body`, using a
per-VM secret at `$HOME/.local/share/dux-amq/amq-secret` (mode 0600).
Receivers verify the magic, freshness (±60 s skew, 24 h window), nonce
dedup, and MAC; reject all four classes silently with stderr warnings.

The wrapper-side mitigation has a structural gap that this issue is
about: AMQ v0.34.0's `--inject-via` runs the external executable with
the wake notification *appended as the final argv element* and does
*not* pipe the message body to the executable's stdin. A receive-side
verifier therefore cannot see the body it needs to verify, and we end
up with two suboptimal options:

  1. Verifier reads stdin → silent under v0.34.0 (no body to check),
     functionally equivalent to dropping all wake notifications.
  2. Verifier ignores stdin and trusts the argv preview → defeats the
     purpose, since the preview is generated by AMQ from the (possibly
     spoofed) message file before any auth check.

Proposed protocol
-----------------
Two-track upstream change so AMQ ships native auth without breaking
existing wake users:

  A. Optional stdin piping for `--inject-via`.
     New flag: `--inject-via-stdin` (default false). When true, AMQ
     pipes the FULL body of each notified message to the executable's
     stdin (one body per invocation, terminated by EOF). The executable
     decides what to do with it (verify, transform, drop). Existing
     argv-preview behavior unchanged.

  B. Native HMAC envelope (lift the dux-amq overlay).
     - `amq init --hmac-secret-path PATH` writes/loads a per-queue
       secret (32 bytes random, mode 0600).
     - `amq send` automatically wraps the body in the DUX1 envelope
       when `meta/config.json` has `"auth": "hmac-v1"`.
     - `amq drain` / `amq wake` automatically verify and unwrap
       envelopes; reject mismatches with a structured error in the
       envelope's `dlq/` directory.
     - Backwards-compat: legacy unsigned messages flagged as
       `verified=false` and only delivered to drain (not wake-inject)
       when `auth = "hmac-v1"` is set.

Threat model recap (out of scope for this issue but worth noting):
the per-VM HMAC protects against a peer with filesystem-write to
`/data/state/amq/` but NOT a peer with full shell as the dux user
(they can read the secret). That's a real escalation gap — see
audit02 T2 in dux-amq-setup/docs/audits/audit02.md — and a future
issue should track moving the secret to a hardware-backed keystore
or running the verifier as a privilege-separated daemon.

Interim mitigation for downstream users (us)
---------------------------------------------
The dux-amq wrappers (claude-amq, codex-amq, gemini-amq) now invoke
`amq wake --inject-via amq-receive-verify` instead of
`--inject-mode raw`. Under AMQ v0.34.0 this means the wake-notification
auto-inject path is silenced (the verifier sees no stdin and exits 0)
but the security goal — *blocking spoofed unsigned messages from being
typed into the agent's TTY* — still holds because no body ever reaches
the TTY through the unsigned path. Agents can continue to drain
verified messages explicitly via `amq drain --include-body` piped
through `amq-receive-verify`. Once track A above lands, the wrappers
will pipe stdin and resume auto-inject for verified messages.

Reproduction (compromised-peer spoof, no auth):
    amq init --root /tmp/q --agents alice,bob --force
    amq send --from alice --to bob --root /tmp/q "evil text"
    # bob's `amq wake --inject-mode raw` types "evil text" into Claude
    # with no way to tell it didn't come from a real alice.

With the proposed `auth = "hmac-v1"` and `--inject-via-stdin`:
    amq init --root /tmp/q --agents alice,bob --auth hmac-v1
    amq send --from alice --to bob --root /tmp/q "valid"  # signed by AMQ
    # bob's wake runs verifier on stdin, body matches MAC, injected.
    # Forged unsigned writes to bob's inbox/ are dropped by the verifier
    # (or by AMQ itself when --auth hmac-v1 is set on the queue).
